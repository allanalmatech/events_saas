**Prompt: Build a Multi-Tenant Event Management SaaS in Core PHP (No MVC)**

Build a **modern, modular, multi-tenant SaaS application** using **Core PHP, MySQL, HTML, CSS, JavaScript, Bootstrap where helpful**, but **strictly no MVC framework**. The system is for **event service providers** such as decorators, tent hire businesses, cake makers, chair hire businesses, sound providers, and other rentable-event-item businesses. The platform must support **multiple tenants**, where each tenant runs their own business independently, but also participates in a **community marketplace and messaging ecosystem** within the SaaS.

## 1. Architecture and General Requirements

The project must be:

* **Core PHP only**
* **Modular file structure**
* **No MVC**
* **MySQL database**
* **Responsive on mobile, tablet, laptop, and large screens**
* **Modern UI with glassmorphism feel**
* **Beautiful icons**
* **Collapsible sidebar with submenu behavior**
* **Theme customization with up to 10 themes**
* **Dark mode included**
* **Default theme: shades of brown**
* **Clean separation of concerns using include files, function files, module folders, CSS files, JS files, and API/action handlers**

The system should feel premium, practical, and business-ready.

## 2. Multi-Tenant SaaS Model

The application must support **tenants**, where each tenant represents a business owner using the system.

### SaaS role hierarchy:

1. **Director of the SaaS**

   * Highest platform-level authority
   * Approves or rejects newly created business-owner accounts
   * Can broadcast messages to super admins
   * Can see all tenants, subscriptions, support tickets, usage, and system analytics
   * Can manage platform-wide settings

2. **Super Admin (Tenant Owner / Business Owner)**

   * Created after approval by the Director
   * Fully controls only their own tenant/business
   * Can add employees/users under their business
   * Can assign permissions to employees using RBAC
   * Can customize business details, branding, themes, and settings
   * Can manage inventory, services, bookings, invoices, receipts, returns, workers, broadcasts, and reports for only their tenant

3. **Employees / Staff under a Tenant**

   * Created by the Super Admin
   * Access only what the Super Admin allows
   * Must have profile management
   * Can upload profile image, phone, contacts, role details, and other personal profile info
   * Their actions must be logged in audit trails

### Tenant isolation:

* One tenant **must never access** another tenant’s internal records directly
* All data must be scoped by `tenant_id`
* Every module must enforce tenant boundaries
* Shared communication between tenants must only happen through the designated marketplace/messaging area

### Example plan logic:

## Y. Account Locking and Enforcement

The Director must be able to **lock tenant accounts** if they fail to pay.

### Locking rules:

* If tenant subscription becomes overdue beyond allowed grace period, account can be restricted
* Director can manually lock/unlock tenant
* System can auto-lock based on unpaid status if enabled
* Locked tenants should not lose data
* Locked tenants should see a clear notice explaining:

  * their plan
  * amount due
  * due date
  * grace period status
  * how to contact support/director

### While locked:

The tenant may be restricted from:

* Creating bookings
* Creating invoices
* Adding new users
* Accessing advanced features
* Using marketplace
* Printing new documents

The director should be able to choose one of two lock modes:

1. **Soft lock**

   * Tenant can log in
   * Can view records
   * Cannot create new operational records
2. **Hard lock**

   * Tenant account access is fully blocked except billing/support page

This must be configurable.
